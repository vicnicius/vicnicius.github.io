Game Development Journey

/Scoping

17-04-2017

I have a long story of unfinished side-projects. I am always full of ideas, and as most developers, I love to learn and use new technologies. This has led me to a cycle of learning, creating, developing, but never finishing projects. I think doing this sometimes is ok. Sometimes the project only serves a learning purpose, and it is fine to just not finish it. With my game project though, I don't want this to be true.

As I have already mentioned, I want to dig into game development so I can express myself in a meaningful way using a form of expression that appeals to me. I want to explore gaming as a form of art. And as in different forms of art, to do so, I need to learn, understand and master the tools and techniques used in the craft. And last but not least, I need to develop my artistic process of creation, so that I can reproduce my steps (that worked) again in the future.

There are lots of tools we can think of when it comes to game development. I've spent some time researching about the issue. If this was just about making a game and delivering it to the public as soon as possible, I would definitely choose one of the mainstream game engines out there, like Unity3D, Unreal Engine 4 or Game Maker. Those tools allow you to build for a variety of platforms, and provide you with lots of out of the box features that help you with the most common issues you would face when developing a game.

But I'm still a web developer. And I know this will be a long journey, so I want to make every step count. As a consequence, when making the decision about the tools I would be willing to learn, I also took into consideration how much learning and using that tool would make me a better web developer... Those tools will be:

  • Haskell, as my programming language
  • GIMP, for visual assets production
  • Audacity, for audio assets production, if needed
  • If I need any more softwares, I will always look for the open source alternatives.

Why Haskell

Given my background as a software developer, maybe the most interesting decision regards the programming language in which I decided to build this game. I've started trying some haskell since we started using a project built with it on Catarse. And from those first experiences, I felt like understanding Functional Programming and doing so inside Haskell's domain made me a better programmer. It influences the way I think about code, even when I was coding on javascript or ruby or php. But studying Haskell can be hard, so it takes a more structured approach to move up on your learning curve and actually be able to write it. With this project I'm starting to take this more formal approach.

Why GIMP, Audacity and other open source tools

Because I believe in open source, and in it's power to change the world. And I also believe that by using open source, contributing to it and sharing related knowledge, we can make technology accessible to more people.

On the other hand, art is one of the most relavant ways a culture might exist in society. Games are a new of culture expression. They are art. So we should make it's production accessible. Open source is one of the best ways to do so.

Next steps

For the next weeks, I'm planning to start learning a bit more about Haskell and game art production.
